Architecture Approaches

Three ways to build chat-browser — an npm-installable local webapp for discovering and searching CLI chat sessions

A

Monolith SPA

Stack: Express backend + React/Vite frontend, bundled together

Single npx chat-browser command starts Express which serves the API and the pre-built SPA. SQLite via better-sqlite3. Chokidar for file-watching. Everything in one process.

Pros

  • Simplest deployment — one process, one port
  • Easy npm install story
  • Shared types between frontend/backend

Cons

  • Heavier npm package (React bundle)
  • Frontend/backend coupled in one repo
  • Harder to swap UI framework later
B

Lightweight Server + HTMX

Stack: Express/Fastify backend, server-rendered HTML with HTMX for interactivity

No build step for frontend. Server renders HTML partials, HTMX handles dynamic updates. Chart.js for analytics. Tiny bundle, fast install.

Pros

  • Minimal dependencies — tiny install footprint
  • No frontend build pipeline
  • Fast to develop, easy to maintain

Cons

  • Less interactive feel than a full SPA
  • Complex UI states harder with HTMX
  • Analytics dashboards may feel limited
C

API Server + Standalone SPA (Recommended)

Stack: Fastify API + React/Vite SPA, pre-built and served statically

Clean separation: Fastify serves a REST API + the pre-built SPA as static files. Frontend is built at publish time (not install time). SQLite + chokidar in the backend. Could later support a CLI-only mode or alternative frontends.

Pros

  • Clean API boundary — testable, extensible
  • SPA gives rich search/filter/analytics UX
  • Pre-built at publish — no build step on install
  • API could power a future CLI search mode

Cons

  • More initial scaffolding than HTMX
  • Larger package than option B
